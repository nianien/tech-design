AI 建设——从场景到平台

0. 摘要

本篇承接《业务篇》的全链路闭环，回答技术侧四个问题：① 各业务环节哪些可由 AI 赋能；② 哪些 AI 能力可跨场景复用；③ 数据飞轮如何在工程上运转；④ AI 建设如何从「场景」演进到「平台」与 AIMOS。

主线判断：AI 不是为每条业务线各造一套，而是先在单场景验证、再把通用能力（内容生成 / RAG / 语义化 / 测评）抽取为共享服务、最终收敛为中台平台。采集层（埋点/管道）是工程前提，AI 真正发力在「语义化 + 测评 + 生成」。

4. 从场景到平台的演变思路

AI 建设不应一上来就建大平台，也不该永远停在零散场景。正确路径是四阶段递进——先在场景里验证有效，再把复用的能力沉淀为服务，收敛为中台平台，最终演进为 Agent 编排的全链路营销 AIMOS。

图 3　AI 建设演变：场景 → 能力 → 平台 → AIMOS

4.1 四个阶段

  • 阶段一 · 场景化（点状 AI）：在单一场景做最小验证（如 DTC 的内容生成、SEO 的可引用单元），目标是「证明有效」与「快速拿数据」。允许各自为战，但要埋好统一的数据主线。
  • 阶段二 · 能力沉淀（共享服务）：当 2–3 个场景开始重复用到同一能力（内容引擎、RAG、语义化、测评），把它从场景里抽取为独立的共享服务，统一接口、统一维护，停止重复造轮子。
  • 阶段三 · 中台平台化：把共享服务收敛为统一的 AI 能力层（模型网关、RAG、语义化、测评、生成），加上治理与接入层（鉴权/版本/监控/Eval/安全），对所有业务线开放。数据飞轮跨场景复利。
  • 阶段四 · 全链路营销 AIMOS：把平台能力包装成 Agent 的工具，客户面向一个 Agent 用对话完成建站、内容运营、SEO、数据分析。交互范式从「人找功能」翻转为「Agent 编排能力」。

4.2 何时从场景走向平台（判据与反模式）

  • 升级判据：当同一能力被 2–3 个场景复用、且接口趋于稳定时，抽取为平台服务（内容生成/RAG/语义化/测评最先满足）。
  • 反模式一·过早平台化：一个场景还没验证就先建大中台——重投入、慢见效，违背快速验证。
  • 反模式二·过晚平台化：多个场景各自重复造内容引擎/RAG——重复建设、数据割裂。
中台边界
AI 中台只做核心 AI 技术能力（内容引擎/RAG/语义化/测评/Agent 框架），不做行业前台产品；前台的行业方案在中台之上构建。这条边界保证平台聚焦、可复用。

5. 阶段四 · 全链路营销 AIMOS

终极形态是把认知翻转过来：不是 AI 为各模块服务，而是各能力作为工具为 Agent 服务。客户面向一个 Agent，用对话交互式完成建站、内容运营、SEO 优化、数据分析。

图 4　范式翻转：从「AI 为模块服务」到「能力为 Agent 服务」

5.1 可行性判断

  • 方向正确：它是「场景→平台」演变的自然终局；已沉淀的能力、RAG、测评飞轮正好成为 Agent 的工具、记忆与判断力，前期投入不浪费。
  • 必须自底向上：Agent 的上限 = 其工具的上限。先有规范、可靠、可组合的能力（平台），Agent 才有得调；Agent-first 而无扎实工具 = 会崩的 demo。
  • 不能一步到全自动：今天的 Agent 不足以无人值守地做高风险、不可逆的操作（对外发布、花钱投放、回复客户）；需按可逆性分级、从 copilot 到 autopilot 渐进。

5.2 自动化分级：从 copilot 到 autopilot

风险 / 可逆性操作Agent 自主度
只读、零风险数据分析、SEO 诊断高（可全自动·起步）
可逆、低风险内容草稿、站点改版预览中（生成 → 人审）
不可逆、高风险对外发布、广告花钱、回复评论低（人审批 + 可回滚 + 审计）

最佳起点 = 数据分析：只读、零外部风险、即时见效——客户问「本周怎么样」，Agent 读数、归因、给建议。再依次推进内容运营 copilot、SEO 诊断，最后才是建站与投放（永远带审批闸）。

5.3 AIMOS 四层架构

层职责对应
交互层Agent：意图理解、对话、编排、记忆战略中的 Agent 框架
工具层能力即工具：建站/内容引擎/SEO·GEO/数据查询/发布/CRM，各有清晰契约阶段三沉淀的平台能力
知识与记忆RAG（企业 + 行业 + 效果知识），Agent 的 grounding 与行业 know-how数据飞轮的产物
评测与护栏测评（判断行动是否有效、自我纠错）+ Guardrail（权限/审批/可回滚/审计）治理与接入层
护城河：为什么是你的 AIMOS，而非通用 Agent
通用 Agent（Manus、Coze 等）做不好出海营销，因为没有垂直数据——不知道这个品类哪种内容带来高询盘、哪种 GEO 结构被引用、目标市场的合规边界。你的 RAG + 效果飞轮，就是让 AIMOS「懂这一行」的芯。Agent 是壳，垂直数据飞轮才是芯。

6. 支撑平台化与 AIMOS 的技术架构

6.1 分层架构（标注 AI / 工程边界）

层职责关键模块归属
R1 曝光层独立站页面、社媒帖子、搜索/AI 可见前端展示 + 埋点工程
R2 发布适配站点发布、SEO/GEO 结构化、社媒适配模板、Schema、UTM工程 + AI
R3 内容中台内容模型、生成、校验、版本、分发内容引擎（核心）AI
R4 数据智能RAG、语义化、测评判优、反哺RAG + 测评（核心）AI
R4 采集埋点、回流、数仓数据管道工程
R5 基础设施模型网关、算力、存储、运维、治理网关、向量库、数仓、Eval/审计工程

6.2 RAG 知识库（两层 + 冷启动）

  • 企业层（私有）：品牌/产品/卖点/合规口径，保证事实正确、口径一致，数据隔离。冷启动靠存量客户资料。
  • 行业/品类层（共享）：品类常识、市场本地化、平台规范，以及飞轮回收长出的「效果知识库」——竞品无法复制的复利资产。

6.3 测评体系（飞轮中枢）

  • 三层：离线（质量/合规/事实一致）、在线（A/B 效果裁决）、语义判优（把效果归因到内容特征）。
  • 基线锚点：「是否带来更高价值的互动/转化」，而非「内容像不像人写」。

7. 落地路径

阶段目标重点
短期 0–3 月（场景化）最小可行飞轮跑通，证明 AI 有用 + 飞轮会学内容引擎 + 语义化 + 测评；DTC 试点、采集借力、约 12 周 Go/No-Go
中期 3–12 月（能力沉淀）扩场景，抽取共享能力接建站转化、CRM 的 AI SDR；把内容引擎/RAG/测评抽成共享服务
长期 12 月+（平台化 → AIMOS）中台平台 + RaaS 飞轮，演进到 Agent 编排统一能力层 + 治理；跨场景复利；按结果计价；Agent 化交互（从数据分析起步）

一句话：从一个会学习的最小飞轮起步，每多复用一个场景就把能力往平台沉淀一层——场景验证价值、平台放大价值。
