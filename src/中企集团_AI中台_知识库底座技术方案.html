AI 中台 · 知识库底座技术与实施方案

服务建站 · 社媒 · GEO/SEO 的统一知识库底座 ｜ 选型 + 工程实施

建站、社媒、GEO/SEO 三类业务都要用到知识——客户画像、行业洞察、品牌口径、产品资料、领域专业知识、过往投放效果。今天这些散在各业务线、各自维护、互不复用。本方案把它们统一到一套知识库底座:用一致的方式获取、清洗、入库、对外服务,做成 AI 中台的记忆与检索层。技术路线:向量检索 + 知识图谱混合、云上托管、国际多语种栈。

一、目标与设计原则

  • 不重复造库:三业务线只建一次、各处复用,口径统一。
  • 知识能互通:同一客户、同一产品的知识,被三业务同时看到,而非各存一份。
  • 给 AI 中台供血:底座是 AI 中台的"记忆 + 检索"层,Agent、内容生成、SEO/GEO 诊断都从这里取知识(grounding)。
原则含义
统一可复用一次入库、多业务调用;能力以服务/接口对外,各线不私接私存。
向量 + 图谱混合语义检索用向量;实体关系、多跳推理用知识图谱,按库取舍。
云上托管优先组件选成熟云服务,降运维、快落地;核心资产保持可迁移。
国际多语种栈出海多语种,嵌入/大模型/解析选国际线、强多语言。
可治理可观测权限、合规、生命周期可审计;运行状态与质量可度量(按需)。
渐进落地先小可用跑通,再扩库、上图谱、补治理。

非目标:不替代客户自有 CRM/ERP,不做通用大模型训练;聚焦"知识的获取→清洗→入库→对外服务"这条链路。

边界:只做底座,不碰业务逻辑
本底座只提供知识能力——获取、清洗、入库,以及对外的检索 / 图谱查询 / RAG(带引用)能力,通过接口与 MCP 工具暴露。它不承载营销决策:像"多图谱联合推理、产出营销策略、营销决策闭环"这类业务逻辑,属于上层业务(如 AIMI 智能基座),由业务侧调用底座来完成——底座负责"存图谱、供图谱检索",业务负责"用图谱做推理与决策"。底座要保证的,是把图谱查询、子图抽取、多跳检索等能力以接口充分暴露,让上层推理"调得动"。分期建设:知识库层(挂载式检索/RAG/图谱)本期建;Agent 层(横向通用 Agent,对外完整 AI 能力)为规划演进——只承载内容生成、GEO 诊断等知识接地、跨业务线可复用的 Agent,纵向营销决策编排仍归业务侧。

二、总体架构

在总体技术分层中的位置
对照公司《技术分层架构图》的五层依赖结构(R1 用户曝光 → R2 发布域/渠道适配 → R3 内容资产中台 → R4 数据智能/知识图谱 → R5 基础设施,上层依赖下层),本方案是第 R4 层(数据智能 / 知识图谱)的实施级展开,并复用 R5 基础设施(模型网关、调度、存储、运维)。R1–R3 是底座的调用方——内容资产中台(R3)同样依赖底座取能力来生产内容资产。下面的"逻辑七层"是对 R4 这一层的内部拆解,与五层依赖图不是同一粒度,勿混淆。
R4 内部再分两子层:知识库层(挂载)+ Agent 层(完整能力)
R4 对外有两种粒度的出口,职责与"谁拥有循环"不同:
  • 知识库层(下,本期重点):提供挂载式能力——检索 / RAG / 图谱,以 MCP/OpenAPI 暴露,可被任意模型或 Agent 挂上去做 grounding。循环在调用方:它要"相关知识",取了自己用。
  • Agent 层(上,规划演进):在知识库层之上提供完整 AI 能力——编排 + 工具调用 + 多步推理,对外是"给目标 → 交付成品"。循环在 Agent:它自己挂载知识库、调工具、产出结果。
边界:Agent 层只放横向、可复用、知识接地的通用 Agent(如内容生成 Agent、GEO 诊断 Agent);纵向的营销策略与决策编排仍归业务侧(如 AIMI 基座),与第一章"多图谱联合推理、营销决策闭环属上层业务"的边界一致。

2.1 逻辑七层

逻辑七层:数据自下而上构建,服务自上而下调用;治理、元数据、可观测、网关四件事横切所有层。

2.2 系统分层架构

把逻辑层落到可部署的组件,按层叠起来看:接入方 → 接入层 → 能力层(含在线服务与离线处理两条)→ 模型与网关 → 存储层,治理 / 安全 / 多租户 / 可观测 / 元数据横切各层。在线服务读存储、离线处理写存储,共享同一套存储与网关。

纯分层架构:接入方 → 接入层(网关) → 能力层(在线服务 / 离线处理) → 模型与网关 → 存储层;治理、安全、多租户、可观测、元数据横切各层。在线服务读存储、离线处理写存储,共享同一套存储。右侧红色虚线为业务回流:调用方产生的效果 / 采纳数据经接入层与离线处理沉淀回效果库(呼应第三章数据飞轮)。

  • API 网关:所有调用的统一入口,负责鉴权、限流、路由,同时暴露 REST 与 MCP 两种协议。
  • 编排服务(知识库层):在线请求的大脑——决定查哪个库、混合检索、重排、组装 RAG 上下文;以 MCP/OpenAPI 暴露挂载式能力,循环在调用方。
  • Agent 层(规划演进):在知识库层之上提供完整 AI 能力(编排 + 工具 + 多步推理),只承载横向通用 Agent(内容生成、GEO 诊断);循环在 Agent,纵向营销决策仍归业务侧。
  • Token/LLM 网关:所有大模型/嵌入调用都经它,统一密钥、计量、限流、缓存、降级。
  • 离线流水线:调度器驱动采集→清洗→构建→入库各 Worker,通过消息队列解耦、可重跑、带状态。
  • 共享存储:向量库/图库/文档库/对象存储 + 主数据(MDM)与 ACL 权限表,在线读、离线写。

三、业务架构:六大知识库与业务关系

3.1 业务线与底座的关系

各调用方与底座是"读能力、写回流"的关系:通过接口/工具读知识能力,产生的数据再回流到效果库。底座只出能力,不碰业务决策。调用方包括三条业务线(建站 / 社媒 / GEO·SEO)、可独立的广告场景,以及总体架构中的内容资产中台(R3)——它依赖底座取 RAG / 检索能力来生产内容资产(依赖方向:上层依赖底座,不反向)。

各调用方向底座读能力(上半蓝箭头:检索 / RAG / 图谱 / Agent 工具);运行后的成效经① 写回流(底部红干线)汇入效果库,效果库再② 反哺优选各库,形成闭环数据飞轮。广告现挂社媒、可独立,内容中台(R3)作为上层同样依赖底座。底座只提供知识能力,营销生成与决策在业务侧。

调用方主要调用能力主要用到的知识库回流到效果库的数据
建站RAG 问答 / 检索产品库、品牌库、客户库页面访问、转化
社媒图谱检索 + 检索 + RAG行业库、效果库、客户库互动、归因、线索
广告(现挂社媒)检索 + RAG产品库、品牌库、效果库广告展现、点击、落地页转化

3.3 六库与营销推理链

按业务推理链看六库:一次营销机会要回答"卖什么→卖到哪→卖给谁→为什么买→怎么表达→效果如何",六个库分别为每一步供给知识(产品/行业/客户/领域/品牌/效果)。横向以 客户 ID、产品 ID 主线串联;效果库把跑出来的成效回流反哺各库,形成数据飞轮。(推理与决策在业务侧,底座只供知识)

3.4 典型场景:底座分别提供什么能力

这里只列底座能提供的能力;怎么用、怎么决策由业务侧。

  • 建站 · 产品详情页
    • 多库过滤检索:按 customer_id + 知识库(产品 / 品牌)过滤,返回相关片段并带出处。
    • 带引用 RAG:把检索到的片段交大模型(经 Token 网关)生成带引用的文案。
    • 前置数据接入:离线链路把客户上传的产品 / 品牌资料解析、切分、入库。
    • 效果回流接口:接收页面访问、转化埋点,写入效果库。
  • 社媒 · 选题发文
    • 图谱多跳查询:在行业图谱上取趋势、竞品关系、关联主题(2 跳子图)。
    • 切片检索:效果库按行业 × 市场切片,返回同类已沉淀的成效数据。
    • 混合检索 + RAG:向量 + 关键词召回素材、重排,带引用生成草稿。
    • 效果回流接口:接收互动、归因、线索数据,写入效果库。
  • 广告 · 落地页 / 素材(现挂社媒,可独立)
    • 卖点 / CTA 检索:按 customer_id + 产品 / 品牌库过滤,取卖点句、合规口径。
    • 行为加权召回:优先返回效果库中高点击 / 高转化的素材与文案版本。
    • 带引用 RAG:生成广告文案 / 落地页卖点,带品牌口径约束。
    • 效果回流接口:接收广告展现、点击、落地页转化与追踪参数,写入效果库。
  • GEO/SEO · 可引用内容
    • 领域 / 行业检索:返回权威、结构化的领域知识与行业事实(带出处)。
    • 可引用单元输出:以结构化、可溯源、易被生成式 AI 引用的格式返回。
    • 采集入库:离线链路爬取 SERP 与 AI 引用情况,入效果 / 行业库。
    • 被引数据查询:按主题 / 内容查询被引情况(含 AI 引用率),支撑被引监控。

共性能力:检索强制按 tenant / customer 过滤(不串户)、所有模型调用经 Token 网关、写入统一沉淀进效果库。

交互的统一原则
业务系统对底座只通过接口读、通过链路写,不碰底层存储。所有读带租户与权限上下文(A 客户看不到 B 客户),所有写最终沉淀进效果库,既隔离了客户数据,又让全公司的实践持续回流到同一个飞轮。回流数据默认归本客户租户;只有经过脱敏、且切片客户数达到 k-匿名阈值的切片,才会进入 shared 库供跨客户复用。

四、数据结构(Schema)

先定架构层的统一存储模型(知识记录、知识图谱、主数据与效果库),它对来源无知;再单独定义核心知识层——以 SSOT 为单一真相、OKF 为导出格式的 concept 模型,以及它如何派生到架构层。

4.1 统一知识记录(chunk)

chunk 表只承载非结构化知识(文档、网页、资料切分后的文本片段),六个库共用一张表(用 kb 区分),便于统一检索与权限。结构化主体不进这张表:客户画像 / 行为走主数据与关系表(4.3 MDM),效果指标走 effect 表(4.3),它们是指标 / 时序型数据,不该硬塞进"chunk + embedding"。需要语义检索时,只把它们的可读摘要(如客户简介、效果洞察小结)生成一条 chunk 切片指回主体,主体留在各自的关系表。统一记录结构:

4.4 核心知识层:SSOT 与 OKF

核心知识(产品 schema、品牌口径、领域术语本体、行业实体、GEO 可引用单元)结构清晰、需人工或 Agent 审校,单独建模。这里有一个必须分清的边界:格式规范 ≠ 系统架构。OKF(Open Knowledge Format)是 Google 提出的可移植知识格式(带 frontmatter 的 markdown、concept 互链成图),它定位是编辑层 / 交换层的约定;底座不把 OKF 文件当作内部真相,而是把真相放在自动化管线写入的结构化存储里。

第一性原理
真相住在 SSOT(单一真相源),所有 markdown / git / 向量 / 图谱 / HTML 表示都是可随时纯函数式重建的派生产物。身份稳定、可 review、可审计、可增量、链接不腐烂,都是这一条的推论。

SSOT:单一真相源(数据库,自动化写入)

底座的知识生产以自动化为主(Agent 生成 concept、enrichment、向量、图谱),不经过 git。真相落在结构化存储中,持有:

SSOT 内容说明
Concept 记录以稳定 ID(主键 / uuid)为主键的结构化记录:frontmatter 字段、正文、summary
图谱边概念间关系直接以 from_id → to_id 的边存在;SSOT 中不存在"路径链接",链接即图的一条边
派生索引源数据向量、图谱索引等属派生层,可随时从 SSOT 重建

身份模型:稳定 ID,与路径解耦

Concept 的身份由 SSOT 中的稳定 ID 承载,从不依附文件路径。"路径即身份"对纯 OKF 编辑层成立,但对企业平台是危险假设——组织调整使目录从 sales/tables/orders 迁到 commerce/tables/orders 时,若路径即身份,则图谱、引用、向量、citation 全部要迁移。锚定到 ID 后,目录的重命名 / 移动 / 拆分 / 合并都不影响身份与引用:正文(及图谱)里的引用直接基于 ID,链接腐烂问题被结构性消除。路径只是 ID 在导出那一刻生成的展示投影。

OKF 的角色:导出 / 交换格式,非内核

本系统内部格式不是 conformant OKF。OKF 的正文链接指向文件路径(保证 GitHub 可点、可移植);系统内部用 ID 图谱边。二者关系是:

  • 系统内部 / SSOT:基于稳定 ID 的图谱边 from_id → to_id。
  • 导出为 conformant OKF:export 时把 ID 边确定性地展开为路径链接,生成可在 GitHub 渲染、可打包流通的标准 bundle。

OKF 因此是底座的出口格式之一,而非真相存储。海量长尾内容(报告 / 新闻 / SERP)既不进 SSOT 也不导出 OKF,直接切块入库;核心知识与长尾内容分两条轨入库。

派生产物:一切人 / 外部可见表示都是 build 产物

五、数据接入与离线链路

5.1 统一数据接入层(多渠道 + 配置化)

数据源不是一两个,而是多渠道、持续接入。底座必须有一层统一数据接入层把所有渠道归一化,并用配置(而非改代码)定义每个数据源怎么接、怎么洗、入哪个库——新增数据源 = 加一份配置。

五类渠道(推 / 拉 / 爬 / 传 / 订)统一汇入接入网关:归一化、打元数据、入队;每个数据源由 Source Registry 的一份配置定义,新增源改配置即可。

渠道方向典型来源接入方式
业务灌入推(push)CRM/建站/社媒的变更、业务主动上报Webhook / 消息队列 / 开放写接口
中台拉取拉(pull)客户数据库、数仓、平台 API连接器(Airbyte)/ CDC / 定时同步
自动爬取爬(crawl)行业站、SERP、竞品、GEO 引用监控托管爬虫(Firecrawl)+ 调度
文件导入传(upload)品牌手册、产品资料、说明书对象存储 + 文档解析
订阅订(subscribe)新闻流、平台事件流订阅连接器 / 流接入

所有渠道进来后统一归一化(打统一元数据、转统一消息格式)、进接入队列,再交给离线流水线。接入层屏蔽渠道差异,上层只看到"标准化的数据进来了"。

注意:这里五类是原始数据源。核心知识(concept / SSOT)不是原始源,而是从这些原始数据加工出来的派生产物(5.3 的"生成 concept"),再作为 RAG 的数据源——它进入的是离线流水线的核心轨,不在本接入网关之列。

配置化:一个数据源一份配置

来源与连接、调度、入哪个库、字段映射、清洗规则、增量键、脱敏、权限,全部声明式定义。新增或调整数据源改配置即可,不动代码。以 CRM 增量拉取为例:

# 从客户 CRM 增量拉取线索 → 客户库
source:
  id: src_crm_leads          # 数据源唯一标识
  type: db                   # db | api | webhook | crawl | file | subscribe
  channel: pull              # push | pull | crawl | upload | subscribe
  conn: ${CRM_DSN}
  schedule: "*/15 * * * *"    # 增量调度
  incremental_key: updated_at
target_kb: customer          # 入哪个知识库
mapping:                     # 字段映射到统一 schema
  customer_id: $.company_id
  text: $.notes
  lang: $.lang
clean_rules:                 # 清洗规则(声明式)
  - { dedup: [customer_id, updated_at] }
  - { normalize: [date, currency] }
  - { entity_extract: [company, person] }
pii: { redact: [email, phone] }
acl: ["role:sales", "tenant:zqdl"]

其他渠道同构,只需改 type / channel 与字段映射:如爬取 SERP 做被引监控,设 type: crawl + seeds,target_kb: effect,mapping 取 snippet / url / region 即可。

接入层把"数据源治理"沉成可配置的资产:谁、从哪、多久、洗什么、入哪、给谁看,全部一目了然、可审计、可复用。

5.2 加工与入库链路

离线入库全流程,三处判断决定走向:内容没变就跳过(省嵌入)、质检不合格进隔离、三处写齐才切版本置 READY;任一 Worker 失败累计超阈值进 ingest.dlq,修复后重放。长尾轨(裸文本)走中间主链,经两道闸门;核心轨(紫线)在解析/清洗后先生成 concept 写入 SSOT,绕开闸门直接进③构建,并以 concept 为单位整体重建。

每个环节是一个独立 Worker,用消息队列解耦、可重跑、带状态——不是一段串行脚本。下面是可照着实现的处理模型。

多模态资产(分期):建站 / 社媒有大量图片(Banner、产品图)与短视频。一期在解析环节做文本化提取——图片走 OCR / caption / alt、视频走转写 / 关键帧标签,产出的文本接入现有文本管线,与其它 chunk 同等检索。演进项:引入多模态嵌入(图文同向量空间),支持以文搜图 / 图文混排检索。

处理模型与队列

  • 队列 topic 串起各环节:ingest.raw →(解析清洗)→ ingest.clean →(构建)→ ingest.write;失败进 ingest.dlq。
  • Worker 无状态、可水平扩展;调度器(Airflow/Dagster)按 source 配置触发拉取类,push 类由 Webhook 直接投递。

环节间消息信封(统一格式)

{ "job_id":"job_77", "source_id":"src_crm_leads", "tenant_id":"t_zqdl",
  "record_id":"rec_998",            // 由 源主键+增量键 确定,幂等
  "kb":"customer", "stage":"clean", "attempt":1,
  "content_hash":"sha256:9af...",   // 归一化正文哈希,用于跳过未变更
  "payload":{ ... }, "ts":"2026-06-10T08:00:00Z" }

状态机(每个 doc/chunk 一条)

discovered → fetched → parsed → cleaned → chunked → embedded → indexed → ready;异常 → quarantined;被新版本取代 → superseded。状态落 pipeline_state 表,断点可续、可重跑。

幂等与增量(省钱的关键)

下面是长尾轨(裸文本切块)的增量机制。

  • record_id 由源主键确定 → 重复投递不会重复入库。
  • 比对 content_hash:正文没变就跳过嵌入与写入——嵌入是主要成本,增量场景绝大多数记录会被跳过。
  • chunk_id = hash(doc_id + 序号 + content_hash) → 向量/图/文档三处都用 upsert,天然幂等。
  • 文档级 version:新版本写新 chunk,全部 indexed 后再把旧版本标 superseded;检索只读 ready 且最新版本,不会读到半成品。

跨存储一致性(不上 XA 分布式事务)

写顺序:文档库 → 向量库 → 图库,三处都是按 chunk_id / node_id 的幂等 upsert,任一步失败可安全重试。用 outbox 表记待写、对账 job 定期核对三处计数差异;最终一致,检索侧靠 version 兜底。

失败与重放

重试带指数退避;超过 N 次进 ingest.dlq + quarantine 表(记原因与原始件指针);修复后一键重新入队重放,不丢数据。

5.3 双轨入库:核心轨(SSOT / concept)与长尾轨

上面这条链路适用于海量长尾内容(报告 / 新闻 / SERP、历史投放原文、行为流)——直接切块入库,不强制结构化。核心知识(产品、品牌、领域、行业实体、GEO 可引用单元)则走另一条轨:提炼成 concept 写入 SSOT,再由 SSOT 派生入库。两条轨共用同一套存储与 chunk 写入口,区别只在前段:

轨链路适用规模
核心轨原始 → 提炼为 concept(写入 SSOT)→ 派生 chunk / 图 / 向量,并导出 OKF 视图领域、产品、品牌、行业实体、GEO 可引用单元小、高价值、SSOT 为真相
长尾轨原始 → 切块 → 向量(即上面 5.2 的链路)海量报告 / 新闻 / SERP、历史投放原文、行为流大、低单值

不全量走核心轨,是因为 OKF / concept 面向数量有限、需人工审校的高价值知识;长尾百万级片段无法逐条精修成 concept,若强制要求,瓶颈会从向量数量转移到精修产能。规模问题由分层处理,核心轨本身不解决规模。下面依次讲核心轨的三件事:怎么生成 concept、存在哪、怎么增量。

一、生成 concept(核心轨前段)

核心知识进入派生入库之前,先经一段提炼环节转成 concept,接在解析 / 清洗之后、构建之前(长尾轨没有这一段):

  • 抽取:抽取 Worker 读清洗后的内容,按 concept 边界拆出实体(一张表 / 一个指标 / 一个产品 / 一条 runbook),为每个生成 concept 草稿,填 frontmatter(type 必填、title、tags、summary)、正文(schema / 卖点 / 步骤)与到其他 concept 的图谱边(由实体识别得出)。草稿可由 LLM 起草、规则补全。
  • 人审(按需):核心知识可经人审——在 git 导出视图上 review、把修订写回 SSOT;低风险或机器维护的可直接入 SSOT 或抽样复核。人审的修订量沉淀为效果库的采纳信号。
  • 写入:定稿 concept 写入 SSOT(稳定 ID 为主键),产出一条变更事件,触发后面的派生入库。

之后的切分 / 嵌入 / 图谱构建与长尾轨共用同一套构建与入库环节,只是输入来自 SSOT 的 concept 而非裸文本。

二、存在哪:写入 SSOT,git / 站点是导出视图

定稿 concept 写入 SSOT(结构化存储),这是唯一真相。供人 review / 阅读 / 交换的 git 仓、HTML 站点、OKF bundle 都是从 SSOT 单向 export 的产物,人不在产物里改知识:

  • 人对 git 视图只读:修改走 SSOT(或经 Agent / 工具),git 永远是单向导出的镜像,不存在"产物反向写回真相"的逆向解析难题。
  • SSOT 变更自动触发导出:"只读"不再是要人遵守的约定,而是系统涌现的事实——任何人的手动 commit 会在下次自动导出时被对齐覆盖。export 须全量、幂等、强制对齐(读 SSOT 全量状态生成应有视图、多余文件删除、改动覆盖回去),对高频写入做去抖 / 合批,commit message 标注 SSOT 版本 / 批次 ID,使 git 视图成为可审计的变更日志。

三、怎么增量:SSOT 变更事件驱动

增量由 SSOT 变更事件驱动:每次 concept 写入产生一条 { concept_id, op } 事件(op ∈ 新增 / 改 / 删 / 改名),与导出后端无关。

  • 最小单位 = concept,整体重建:单个 concept 约几百至几千 token,重建成本低。新增 / 改时删除该 concept 的全部派生向量(doc + section),重新生成、嵌入并 upsert,不在内部逐块求差。
  • 改名零成本:身份是稳定 ID、与路径解耦,改名只是 export 时路径投影变化,不触发任何重嵌——这是 ID 身份相对"路径即身份"的直接收益。
  • 删除与失效传播:删除 concept 时,依赖它的概念由 backlink 索引查出并校验链接完整性;被引概念按 deprecation 处理(标记、检索降权),不硬删。因身份不依附路径、正文不含路径,失效传播主要发生在 export 渲染阶段,真相层基本无腐烂。
  • 图增量:frontmatter 更新为节点属性(无嵌入);图谱边随 SSOT 的 from_id → to_id 变更做差集(增 / 删边),不重建整图。
  • 原子与幂等:变更 concept 全部 upsert 后再推进检索侧的 indexed_ver,读一致快照;按 (concept_id, content_hash) upsert,重放为幂等空操作;失败进 DLQ 重放。派生层可随时从 SSOT 全量重建。

因为身份是稳定 ID、正文不含路径,链接腐烂与跨 bundle 路径反解析在写入阶段就不会发生,无需额外的 resolver 或注册表兜底。跨 bundle 引用就是图谱里一条跨域的边,处理方式与域内边相同。

六、在线数据链路与服务协议

6.1 三级能力,逐层解耦

底座对外的能力按粒度分三级、相互解耦,调用方按需取用——"循环"逐级从调用方转移到底座:

  • ① 检索 / RAG 召回层(挂载式,独立可用):向量 + 图谱 + 混合检索 + 重排,输入查询、输出相关知识片段或子图(带出处)。业务可以只要这一层:取片段自行使用,或挂给自有模型做 grounding。对应 /v1/search、/v1/graph/query。循环在调用方。
  • ② LLM 集成层(可选,靠近业务):在召回结果上接 LLM 做问答/生成,模型走 Token 网关。对应 /v1/rag/query。底座提供"开箱即用的 RAG 问答"作为便利能力,但业务也可自带模型、自行编排。
  • ③ Agent 层(完整能力,规划演进):编排 + 工具调用 + 多步推理,对外"给目标 → 交付成品",循环在 Agent(它自己挂载 ①②、调工具、产出)。只承载横向通用 Agent(内容生成、GEO 诊断);纵向营销决策仍归业务侧。

换模型、加模型或不用模型都不影响检索层。检索是本期的核心能力,模型集成与 Agent 层在其之上,可插拔、可后续演进。

6.2 在线检索时序

一次查询的处理主线:鉴权 → 缓存 → 嵌入 → 并行召回 + 融合 →(关系类按需走图谱)→ 重排 + 行为加权 → 按 mode 返回。几个要点:鉴权后用含 tenant_id 的缓存键命中即返回,杜绝跨客户串户;召回阶段向量与关键词并行、都下推租户与 ACL 过滤;核心知识与长尾内容走同一套接口、落同一索引,来源差异对调用方不可见,只是核心知识(concept)的向量召回多一层"先定位概念、再下钻 section"的结构;最后按 mode 决定只返回检索结果还是接生成层产出带引用的答案,引用统一指向稳定的 concept ID 或原始 source。

实现细节(融合、图谱、两层召回、行为加权、延迟)

混合检索融合(RRF)

向量 top50 与 BM25 top50 用倒数排名融合:每个文档得分 = Σ 1 / (k + 该路命中排名),k 取 60;融合后取前 30 交 rerank,再取前 8 进上下文。向量管语义近、BM25 管关键词准,互补。

核心知识的两层召回

concept 派生 Document 向量与 Section 向量。Document 向量由 title + frontmatter + summary + 各 section 标题 + 从正文确定性抽取的关键词 / 实体生成(不只用 summary,避免"支持 Oracle 吗"这类细节在摘要里被吞掉而漏召回),所有输入从 concept 内容确定性派生、可纯函数重建。召回先用 Document 向量定位相关概念,再在命中 concept 内用 Section 向量精召回;frontmatter(type / tags)与图谱关系作为结构化条件一并下推过滤。两条约束:summary 等派生输入须持久化进 SSOT(否则向量层依赖会漂移、无法重建的数据);防第一层漏召回——概念数数百到数千时第一层以结构化过滤为主、可放宽 Top-K,概念数大时另开 section 直召旁路做 hybrid 融合。Agent 也可不走检索,直接从某 concept 顺图谱边逐层展开读取。

图谱子图

关系类问题先把查询实体链接到图谱,再用 Cypher 拉 2 跳子图(强制带租户过滤):

MATCH (c:Company {id:$ent})-[:IN_INDUSTRY]->(i:Industry)
OPTIONAL MATCH (c)-[:COMPETES_WITH]->(x:Company)
WHERE c.tenant_id = $tenant
RETURN c, i, collect(x)[..20] AS competitors

行为加权

语义重排只解决"相关"、不解决"有效"。效果库的成效信号(点击 / 完读 / 转化 / AI 引用)与采纳信号回灌为排序偏好:最终序 = 语义重排分 × 行为权重因子(按 content_id 关联、按行业 × 市场切片取分位),高转化高引用内容上浮;新内容无流量时用 CMS 采纳信号(原样发布=轻正向、大改=轻负、驳回=过滤)做冷启动先验,权重低于实测成效;权重设上下限防马太效应。加权因子离线更新、在线只读,不增加延迟。

延迟预算(现实口径)

嵌入与重排是外部 API(Cohere),按网络往返(RTT)算:查询改写默认关闭(开则 +150~600ms),嵌入 60~150ms、向量+BM25 并行 30~50ms、图谱 2 跳 40~80ms、重排 100~300ms(路径最大头)。分档目标:纯检索(不 rerank)P95 ≤ 150ms;带外部 rerank P95 ≤ 400ms;RAG 端到端再加生成 1~2.5s。想压到 150ms 且带 rerank,需把嵌入 / rerank 换成自托管或同区低延迟模型。

七、Token / LLM 网关

大模型与嵌入调用都经网关。底层复用公司已有的新网 xapi,底座只加一层薄中间件做应用维度的计量与治理,不重复造网关。

7.1 两层网关:复用底层 + 应用治理

公司已有新网 xapi 模型网关:一个 Key 接 OpenAI / Gemini / Claude 兼容的多模型(含 embeddings、rerank),统一鉴权与平台计费。但它是基础设施级、账户维度的——只到账户出账,分不清某次调用是哪个租户 / 业务 / 会话花的。底座因此不重复造模型接入,而是复用 xapi 做底层,在其之上加一层"应用维度网关",补齐它不做的事。

层由谁提供职责
底层模型接入复用新网 xapi(已有)统一接多模型(文本 / 嵌入 / 重排 / 图像…)、密钥托管、兼容协议、平台(账户级)计费
应用维度治理(薄中间件)底座新增按租户 / 业务 / 会话计量 token 与成本、配额限流、语义 + 结果缓存、按任务降级、调用审计 / PII 脱敏、Prompt 模板管理与版本(对应 R5 大模型的 Prompt 管理)
为什么不再上一个完整网关(避免与 xapi 重复)
LiteLLM / Portkey 这类网关本身也做"统一多模型接入 + 路由 + 密钥",而这部分 xapi 已经做了——再叠一个就是"网关套网关",接入层重复。底座缺的只是 xapi 不做的应用维度计量与治理,所以这层做成一层薄中间件即可:优先自建(在编排服务里包一层,给每次 xapi 调用打 tenant/app/session 标签、从回包 usage 记 token,做配额与缓存);或退一步用 LiteLLM,但只把上游设成 xapi 这一个,纯借它的虚拟 Key / 预算 / 缓存,而不让它再做多厂商接入。

配置示例(应用层薄中间件,上游就是 xapi):

gateway:
  upstream: xinnet-xapi          # 底层模型接入,复用现有网关
  routes:
    - { model: cohere-embed-v4, use: embedding }
    - { model: claude-*, use: generation, fallback: gpt-* }
  metering: { by: [tenant, app, scene, session, task], record: tokens+cost }   # ← 多维计量,xapi 不做
  rate_limits: { tenant_default: { rpm: 600, tpm: 200000 } }
  budget:      { tenant: { monthly_usd: 5000, alert_at: 0.8 } }
  cache:       { semantic: true, ttl: 86400 }
  audit:       { log: all, pii_redact: true }

注:所选模型(如 Cohere Embed v4)若 xapi 暂未覆盖,应用层网关可直连对应厂商,或替换为 xapi 已提供的等价模型——路由对业务透明,不影响上层。

7.2 Token 经济(成本杠杆)

大模型与嵌入是主要可变成本。网关层把成本做到"按租户可见、按杠杆可控",主要省钱杠杆:

杠杆怎么省
增量跳过(content_hash)正文没变不重嵌入——离线最大头成本,增量场景绝大多数被跳过
语义 + 结果缓存重复/近似问答直接命中,不再调用嵌入与生成
按任务路由模型简单抽取/分类用小而便宜的模型,复杂生成才上大模型
重排压上下文rerank 后只喂 top8 而非 top50,生成 token 大幅下降
批量嵌入离线嵌入批处理,降单位成本
预算与配额按租户设月度预算与限流,到阈值告警/降级,杜绝失控

应用维度网关逐次记录 token 与费用,按租户 / 业务线出账——这正是新网 xapi(账户级)不做、需在应用层补上的;它既是成本控制,也是 RaaS 计价时的成本依据。

7.3 网关数据:计量、配额、缓存

计量按五个维度、由粗到细:租户 → 应用 → 场景 → 会话 → 任务。每条用量记录都带全链标签,于是可在任一层级聚合(roll-up),也可在多级同时设限(如:租户月度预算 + 应用日配额 + 会话/任务硬上限)。

这层是有状态的——用量要落库、配额要实时更新。表不多,但要建:

表 / 存储存什么更新时机选型
用量明细 usage每次调用带全维度标签:tenant / app / scene / session / task(trace)+ 模型 / op / prompt+completion tokens / 成本 / 延迟 / 是否命中缓存每次调用后追加写(append-only)Postgres 分区表(量大可转数仓聚合)
配额预算 budget多级:level(tenant/app/scene/session/task)+ 对象 + 预算/RPM/TPM 上限 + 实时已用每次调用后按各级累加 used;周期重置Postgres(强一致,实时累加)
缓存 cache语义缓存(查询向量→结果)、结果缓存(精确键→结果)。两类缓存键都必须并入 tenant_id + ACL:语义缓存按 tenant 分桶(只在同租户向量空间内做近似命中),严禁跨租户共享条目——否则两个客户的近似查询会串户泄漏命中即读、未命中写,带 TTLRedis(向量近似 + KV)
虚拟 Key(可选)租户 / 业务的 Key、权限范围、状态签发 / 吊销时更新Postgres

存储归位:这些是运行 / 计量元数据,落在存储层的 Postgres(usage / budget / vkey)与 Redis(cache),与知识数据分开管理。

几个选型要点

  • 自建 vs 云:基础设施(向量库、图库、解析、嵌入)优先云托管;但知识图谱 / 效果库、入库 pipeline 逻辑两类核心资产必须自有、可迁移——借云的算力和组件,不把飞轮交给云。
  • 图谱分期、按需投入:多数企业问答是查资料 / 规则 / 产品 / FAQ,关系类多跳占比不高,不宜一上来就重投。一期只建图谱 schema + 存储 + concept 投影(从 SSOT 派生,近零成本),检索主力是向量 + BM25 + rerank,同时埋点统计关系类查询真实占比;二期占比与价值显著时,再叠加按重要性采样的图谱抽取与多跳推理。先建 schema 让二期能直接加码,不必重做数据模型。
  • 图谱抽取的成本边界:GraphRAG 索引阶段是重 LLM 调用,千万级 chunk 全量跑成本可达嵌入的数倍。故只对需要多跳的库(行业 / 领域 / 品牌)按重要性采样抽取(高价值文档,长尾走纯向量),抽取调用经 Token 网关计量纳入预算。
  • Query 重写单列:业务查询常口语、含糊或过长,检索前改写 / 扩展 / 拆子查询(必要时 HyDE)可提升召回。它是可插拔前置步骤,失败回退原始 query,不影响主链路。

9.1 容量与性能估算

9.2 非功能要求

  • 可用性:在线服务目标 SLA 99.9%;离线流水线允许有延迟,但必须可重跑、不丢数据。
  • 扩展性:在线服务无状态、水平扩展;向量库按租户/库分片;离线 Worker 按队列堆积自动扩缩。
  • 容灾备份:向量/图/文档库定期快照,对象存储跨区冗余;主数据(MDM)与 ACL 权限表优先级最高、独立备份;明确 RTO/RPO 目标。

9.3 多租户:两根轴分开看

这是全方案最容易混淆、也最影响权限正确性的一点,先定义清楚。底座有两根正交的轴,不要用一根 tenant_id 同时承担:

轴字段含义与规则
隔离轴(数据归谁)tenant_id = 终端客户数据按终端客户硬隔离,A 客户的数据不可能进入 B 客户的结果。每条记录强制带 tenant_id,检索时强制下推过滤。
访问轴(谁来取)app / role建站 / 社媒 / GEO 是带角色的访问方,不是租户。同一终端客户的知识,可被三条业务线跨业务线共享(这正是"知识互通"目标)——共享发生在同一 tenant 内、按 role 控可见,不是跨 tenant。

隔离按客户(tenant),复用按业务线(app/role):"建站和社媒共享同一客户的产品库"是同租户内的事,"A 客户看不到 B 客户"是跨租户的事,两者在不同轴上,不冲突。

  • 隔离实现:向量库用 namespace=tenant、文档库用行级 tenant_id、图库用按 tenant 分子图 + 服务层强制注入;默认逻辑隔离,大客户 / 敏感客户可升级为物理隔离(独立索引 / 命名空间 / 库)。
  • 跨业务线共享:同一 tenant 下,记录的 acl 标 role:builder/social/geo 决定哪条业务线可见;不打 role 标的默认全业务线可见。业务线在网关注入 app + role,服务层据此过滤。
  • 跨客户复用(唯一例外):只有脱敏后的领域库、效果库(按行业 × 市场切片、且满足 9.4 的 k-匿名阈值)落在保留租户 shared 下,所有业务线只读;原始客户数据永不进 shared。
  • 图库隔离(最弱面,重点加固):图查询不暴露裸 Cypher,只暴露参数化子图模板,租户过滤由服务层强制注入(调用方无法绕过);每个 Cypher 模板都内建 WHERE n.tenant_id = $tenant,经评审固化。kb_graph 工具(6.4)同理——只收模板名 + 参数,不收自由查询语句。
  • 配额:网关按 tenant 限流(RPM/TPM)+ 存储配额,单客户打满不影响他人。
  • 写入即归属:接入时每条数据就打 tenant_id + acl,从源头杜绝"无主数据"。

9.4 增量更新与一致性

  • 变更捕获:CRM/产品库等结构源用 CDC/Webhook 近实时触发,外部内容定时增量,只处理变化部分。
  • 原子切换:受影响 chunk 重新嵌入后写新版本,就绪再切换指针,旧版本保留以便回滚——检索侧不会读到半成品。
  • 写一致性:"文档库 + 向量库 + 图库"三写带补偿重试,最终一致 + 版本号兜底,失败可重放。
  • 失效下线:valid_to 到期或被标记失效的知识,检索时过滤、后台定期清理,避免旧知识误导。
  • 跨客户复用的 k-匿名:效果库切片进 shared 供跨客户复用前,必须 anonymized=true 且 cohort_size ≥ k(默认 k=5);切片客户数过少时聚合值仍可能被反推,低于阈值的切片只在本租户内可用,不对外。

9.5 冷启动

  • 新客户(无专属知识/无效果):先用行业库 + 领域库兜底(通用底料),客户库随交互逐步积累。
  • 新行业:先靠领域库的通用方法论,叠加快速采集该行业公开资料建"行业库种子"。
  • 效果库冷启动:没有本客户实测前,先借同行业×市场的切片经验做参考,跑出数据后切回本客户真实效果。

9.7 治理与安全

  • 权限与安全:按租户/客户/角色分级,ACL 写在每条记录;跨客户复用只允许脱敏后的领域、效果知识。
  • 合规与数据驻留:出海数据按区域驻留、按授权使用,过程可审计。
  • 生命周期:知识带来源、时效、版本;过期下线或降权。

9.8 评测:让"越用越准"可度量

RAG 底座的调优(切分粒度、嵌入模型、是否 rerank、图谱是否介入)必须靠评测驱动,否则改了不知好坏、也发现不了回归。评测与治理同级,作为横切能力建设——一期先做离线抽样,以 Ragas 框架建立基线,不必上全自动。
