AI 中台 · 知识库底座技术与实施方案

服务建站 · 社媒 · GEO/SEO 的统一知识库底座 ｜ 选型 + 工程实施

建站、社媒、GEO/SEO 三类业务都要用到知识——客户画像、行业洞察、品牌口径、产品资料、领域专业知识、过往投放效果。今天这些散在各业务线、各自维护、互不复用。本方案把它们统一到一套知识库底座:用一致的方式获取、清洗、入库、对外服务,做成 AI 中台的记忆与检索层。全文沿总体架构 → 业务架构 → 数据结构 → 数据链路 → 服务协议 → 网关 → 工具选型 → 非功能 → 治理展开,既讲设计也给可落地的 Schema、接口与配置。技术路线:向量检索 + 知识图谱混合、云上托管、国际多语种栈。

一、目标与设计原则

建站、社媒、GEO 各自都要用知识,可这些知识今天散在各条线、互不相通。先说清为什么要把它们统一成一套底座,以及几条设计原则和边界。

  • 不重复造库:三业务线只建一次、各处复用,口径统一。
  • 知识能互通:同一客户、同一产品的知识,被三业务同时看到,而非各存一份。
  • 给 AI 中台供血:底座是 AI 中台的"记忆 + 检索"层,Agent、内容生成、SEO/GEO 诊断都从这里取知识(grounding)。
原则含义
统一可复用一次入库、多业务调用;能力以服务/接口对外,各线不私接私存。
向量 + 图谱混合语义检索用向量;实体关系、多跳推理用知识图谱,按库取舍。
云上托管优先组件选成熟云服务,降运维、快落地;核心资产保持可迁移。
国际多语种栈出海多语种,嵌入/大模型/解析选国际线、强多语言。
可治理可观测权限、合规、生命周期可审计;运行状态与质量可度量(按需)。
渐进落地先小可用跑通,再扩库、上图谱、补治理。

非目标:不替代客户自有 CRM/ERP,不做通用大模型训练;聚焦"知识的获取→清洗→入库→对外服务"这条链路。

边界:只做底座,不碰业务逻辑
本底座只提供知识能力——获取、清洗、入库,以及对外的检索 / 图谱查询 / RAG(带引用)能力,通过接口与 MCP 工具暴露。它不承载营销决策:像"多图谱联合推理、产出营销策略、营销决策闭环"这类业务逻辑,属于上层业务(如 AIMI 智能基座),由业务侧调用底座来完成——底座负责"存图谱、供图谱检索",业务负责"用图谱做推理与决策"。底座要保证的,是把图谱查询、子图抽取、多跳检索等能力以接口充分暴露,让上层推理"调得动"。模型 / Agent 编排底座当前不建(编排目前与业务耦合),列为未来演进项。

二、总体架构

先看整体结构:底座逻辑上分七层(自下而上构建知识、自上而下提供服务);落到组件,则是一套自接入方到存储层、横切治理与网关的分层架构。下面分逻辑七层与系统分层架构两部分看。

2.1 逻辑七层

逻辑七层:数据自下而上构建,服务自上而下调用;治理、元数据、可观测、网关四件事横切所有层。

2.2 系统分层架构

把逻辑层落到可部署的组件,按层叠起来看:接入方 → 接入层 → 能力层(含在线服务与离线处理两条)→ 模型与网关 → 存储层,治理 / 安全 / 多租户 / 可观测 / 元数据横切各层。在线服务读存储、离线处理写存储,共享同一套存储与网关。

纯分层架构:接入方 → 接入层(网关) → 能力层(在线服务 / 离线处理) → 模型与网关 → 存储层;治理、安全、多租户、可观测、元数据横切各层。在线服务读存储、离线处理写存储,共享同一套存储。

  • API 网关:所有调用的统一入口,负责鉴权、限流、路由,同时暴露 REST 与 MCP 两种协议。
  • 编排服务:在线请求的大脑——决定查哪个库、混合检索、重排、组装 RAG 上下文。
  • Token/LLM 网关:所有大模型/嵌入调用都经它(详见第七章),统一密钥、计量、限流、缓存、降级。
  • 离线流水线:调度器驱动采集→清洗→构建→入库各 Worker,通过消息队列解耦、可重跑、带状态。
  • 共享存储:向量库/图库/文档库/对象存储 + 主数据(MDM)与 ACL 权限表,在线读、离线写。

三、业务架构:六大知识库与业务关系

这一章从业务侧看底座,回答"底座为业务提供了什么":谁在用、用哪些知识、知识之间怎么关联、典型场景里底座能提供哪些能力。

3.1 业务线与底座的关系

各业务线与底座是"读能力、写回流"的关系:业务线通过接口/工具读知识能力,产生的数据再回流到效果库。底座只出能力,不碰业务决策。

三条业务线向底座"读能力"(检索 / RAG / 图谱 / Agent 工具),把成效"写回流"到效果库;底座只提供知识能力,营销生成与决策在业务侧。

业务线主要调用能力主要用到的知识库回流到效果库的数据
建站RAG 问答 / 检索产品库、品牌库、客户库页面访问、转化
社媒图谱检索 + 检索 + RAG行业库、效果库、客户库互动、归因、线索
GEO / SEO检索 + 图谱(可引用单元)领域库、行业库被引情况、SERP 表现

3.2 六大知识库

知识库存什么主要数据源用途 / 场景检索方式
客户库画像、行为、偏好、互动与成交、所属行业/市场CRM、建站/社媒后台、埋点千人千面、获客诊断、Agent 记忆向量 + 结构化(主键 客户ID)
行业库行业格局、产业链、政策、趋势、竞品行业报告、新闻、竞品站、SERP洞察、选题、GEO 立意向量 + 图谱
品牌库定位、口径、调性、话术、禁区、视觉规范客户资料、官网、过往物料统一品牌口径、防跑偏向量 +(轻)图谱
产品库参数、卖点、FAQ、场景、价格政策客户产品库、官网、说明书、电商 API详情页生成、问答、SEO 结构化向量 + 结构化

3.3 六库与营销推理链

按业务推理链看六库:一次营销机会要回答"卖什么→卖到哪→卖给谁→为什么买→怎么表达→效果如何",六个库分别为每一步供给知识(产品/行业/客户/领域/品牌/效果)。横向以 客户 ID、产品 ID 主线串联;效果库把跑出来的成效回流反哺各库,形成数据飞轮。(推理与决策在业务侧,底座只供知识)

3.4 三个场景:底座分别提供什么能力

这里只列底座能提供的能力;怎么用、怎么决策由业务侧(接口契约见第六章、离线链路见第五章)。

  • 建站 · 产品详情页
    • 多库过滤检索:按 customer_id + 知识库(产品 / 品牌)过滤,返回相关片段并带出处。
    • 带引用 RAG:把检索到的片段交大模型(经 Token 网关)生成带引用的文案。
    • 前置数据接入:离线链路把客户上传的产品 / 品牌资料解析、切分、入库。
    • 效果回流接口:接收页面访问、转化埋点,写入效果库。
  • 社媒 · 选题发文
    • 图谱多跳查询:在行业图谱上取趋势、竞品关系、关联主题(2 跳子图)。
    • 切片检索:效果库按行业 × 市场切片,返回同类已沉淀的成效数据。
    • 混合检索 + RAG:向量 + 关键词召回素材、重排,带引用生成草稿。
    • 效果回流接口:接收互动、归因、线索数据,写入效果库。
  • GEO/SEO · 可引用内容
    • 领域 / 行业检索:返回权威、结构化的领域知识与行业事实(带出处)。
    • 可引用单元输出:以结构化、可溯源、易被生成式 AI 引用的格式返回。
    • 采集入库:离线链路爬取 SERP 与 AI 引用情况,入效果 / 行业库。
    • 被引数据查询:按主题 / 内容查询被引情况,支撑被引监控。

共性能力:检索强制按 tenant / customer 过滤(不串户)、所有模型调用经 Token 网关、写入统一沉淀进效果库。

交互的统一原则
业务系统对底座只通过接口读、通过链路写,不碰底层存储。所有读带租户与权限上下文(A 客户看不到 B 客户),所有写最终沉淀进效果库——这样既隔离了客户数据,又让全公司的实践不断喂回同一个飞轮。

四、数据结构(Schema)

上一章说清了"有哪些知识";要让它们能被检索和复用,先得有统一的数据结构。这里给出入库的核心 Schema:知识记录、知识图谱、主数据与效果库。

五、数据接入与离线链路

知识从源头变成可检索的资产,靠数据接入层和离线流水线。下面讲多渠道、配置化的接入,以及采集 → 清洗 → 构建 → 入库这条链路。

5.1 统一数据接入层(多渠道 + 配置化)

数据源不是一两个,而是多渠道、持续接入。底座必须有一层统一数据接入层把所有渠道归一化,并用配置(而非改代码)定义每个数据源怎么接、怎么洗、入哪个库——新增数据源 = 加一份配置。

五类渠道(推 / 拉 / 爬 / 传 / 订)统一汇入接入网关:归一化、打元数据、入队;每个数据源由 Source Registry 的一份配置定义,新增源改配置即可。

渠道方向典型来源接入方式
业务灌入推(push)CRM/建站/社媒的变更、业务主动上报Webhook / 消息队列 / 开放写接口
中台拉取拉(pull)客户数据库、数仓、平台 API连接器(Airbyte)/ CDC / 定时同步
自动爬取爬(crawl)行业站、SERP、竞品、GEO 引用监控托管爬虫(Firecrawl)+ 调度
文件导入传(upload)品牌手册、产品资料、说明书对象存储 + 文档解析
订阅订(subscribe)新闻流、平台事件流订阅连接器 / 流接入

所有渠道进来后统一归一化(打统一元数据、转统一消息格式)、进接入队列,再交给离线流水线。接入层屏蔽渠道差异,上层只看到"标准化的数据进来了"。

配置化:一个数据源一份配置

来源与连接、调度、入哪个库、字段映射、清洗规则、增量键、脱敏、权限,全部声明式定义。新增或调整数据源改配置即可,不动代码。两个示例:

# 示例一:从客户 CRM 增量拉取线索 → 客户库
source:
  id: src_crm_leads          # 数据源唯一标识
  type: db                   # db | api | webhook | crawl | file | subscribe
  channel: pull              # push | pull | crawl | upload | subscribe
  conn: ${CRM_DSN}
  schedule: "*/15 * * * *"    # 增量调度
  incremental_key: updated_at
target_kb: customer          # 入哪个知识库
mapping:                     # 字段映射到统一 schema
  customer_id: $.company_id
  text: $.notes
  lang: $.lang
clean_rules:                 # 清洗规则(声明式)
  - { dedup: [customer_id, updated_at] }
  - { normalize: [date, currency] }
  - { entity_extract: [company, person] }
pii: { redact: [email, phone] }
acl: ["role:sales", "tenant:zqdl"]
# 示例二:自动爬取 SERP/AI 引用做被引监控 → 效果库
source:
  id: src_geo_serp
  type: crawl
  channel: crawl
  seeds: ["https://serp/...&q=工业连接器"]
  schedule: "0 2 * * *"
target_kb: effect            # GEO 被引监控写入效果库
mapping: { text: $.snippet, source_url: $.url, region: $.gl }
clean_rules: [ { denoise: boilerplate } ]

接入层把"数据源治理"沉成可配置的资产:谁、从哪、多久、洗什么、入哪、给谁看,全部一目了然、可审计、可复用。

5.2 加工与入库链路

离线入库全流程,三处判断决定走向:内容没变就跳过(省嵌入)、质检不合格进隔离、三处写齐才切版本置 READY;任一 Worker 失败累计超阈值进 ingest.dlq,修复后重放。

每个环节是一个独立 Worker,用消息队列解耦、可重跑、带状态——不是一段串行脚本。下面是可照着实现的处理模型。

处理模型与队列

  • 队列 topic 串起各环节:ingest.raw →(解析清洗)→ ingest.clean →(构建)→ ingest.write;失败进 ingest.dlq。
  • Worker 无状态、可水平扩展;调度器(Airflow/Dagster)按 source 配置触发拉取类,push 类由 Webhook 直接投递。

环节间消息信封(统一格式)

{ "job_id":"job_77", "source_id":"src_crm_leads", "tenant_id":"t_zqdl",
  "record_id":"rec_998",            // 由 源主键+增量键 确定,幂等
  "kb":"customer", "stage":"clean", "attempt":1,
  "content_hash":"sha256:9af...",   // 归一化正文哈希,用于跳过未变更
  "payload":{ ... }, "ts":"2026-06-10T08:00:00Z" }

状态机(每个 doc/chunk 一条)

discovered → fetched → parsed → cleaned → chunked → embedded → indexed → ready;异常 → quarantined;被新版本取代 → superseded。状态落 pipeline_state 表,断点可续、可重跑。

幂等与增量(省钱的关键)

  • record_id 由源主键确定 → 重复投递不会重复入库。
  • 比对 content_hash:正文没变就跳过嵌入与写入——嵌入是主要成本,增量场景绝大多数记录会被跳过。
  • chunk_id = hash(doc_id + 序号 + content_hash) → 向量/图/文档三处都用 upsert,天然幂等。
  • 文档级 version:新版本写新 chunk,全部 indexed 后再把旧版本标 superseded;检索只读 ready 且最新版本,不会读到半成品。

跨存储一致性(不上 XA 分布式事务)

写顺序:文档库 → 向量库 → 图库,三处都是按 chunk_id / node_id 的幂等 upsert,任一步失败可安全重试。用 outbox 表记待写、对账 job 定期核对三处计数差异;最终一致,检索侧靠 version 兜底。

失败与重放

重试带指数退避;超过 N 次进 ingest.dlq + quarantine 表(记原因与原始件指针);修复后一键重新入队重放,不丢数据。

六、在线数据链路与服务协议

知识入库后怎么对外用,看在线服务这一层:检索与生成的解耦、一次查询的处理过程,以及对外的 REST / MCP 契约。

6.1 检索与生成,两层解耦

底座把"检索"和"生成"分两层、相互解耦——这点很重要:

  • 检索 / RAG 召回层(独立可用):向量 + 图谱 + 混合检索 + 重排,输入查询、输出相关知识片段或子图(带出处)。业务可以只要这一层——取片段自己用,或喂给自己的模型。对应 /v1/search、/v1/graph/query。
  • LLM 集成层(可选,靠近业务):在召回结果上接 LLM 做问答/生成,模型走 Token 网关。对应 /v1/rag/query。底座提供"开箱即用的 RAG 问答"作为便利能力,但业务也可自带模型、自行编排。

关键:换模型、加模型、甚至不用模型,都不影响检索层。检索是底座的硬能力,模型集成是其上的可插拔层。

6.2 在线检索时序

编排逻辑(对应上图)

  • 鉴权与缓存:先校验租户与知识库权限(无权限直接拒绝);用"归一化查询 + 知识库 + 过滤条件"算缓存键,命中则直接返回,不再检索与生成。
  • 查询改写与嵌入:对原始查询做改写/扩展(失败回退原查询),再经 Token 网关生成查询向量。
  • 并行召回 + 融合:向量检索与关键词(BM25)各取 top50、并行执行(都下推租户与 ACL 过滤),用 RRF 融合成候选集。
  • 图谱按需介入:仅当问题属于关系 / 趋势 / 对比类时,链接实体、取 2 跳子图,合并进候选。
  • 重排与出口:对候选重排后取 top8;mode=search 直接返回检索结果,mode=rag 则再经生成层产出带引用的答案。最后写缓存返回。

混合检索怎么融合(RRF)

向量 top50 与关键词(BM25)top50 用倒数排名融合合并:每个文档得分 = Σ 1 / (k + 该路命中排名),k 取 60;融合后取前 30 交 rerank,再取前 8 进上下文。向量管"语义近"、BM25 管"关键词准",互补。

图谱子图怎么取

关系类问题先把查询里的实体链接到图谱,再用 Cypher 拉 2 跳子图(强制带租户过滤):

MATCH (c:Company {id:$ent})-[:IN_INDUSTRY]->(i:Industry)
OPTIONAL MATCH (c)-[:COMPETES_WITH]->(x:Company)
WHERE c.tenant_id = $tenant
RETURN c, i, collect(x)[..20] AS competitors

延迟预算(并行)

嵌入 ≈ 20ms(命中缓存为 0)· 向量 ≈ 30ms · BM25 ≈ 20ms(与向量并行)· 重排 ≈ 40ms · 图谱 ≈ 50ms(仅关系类走)。检索整体 P95 目标 ≤ 150ms;走生成时再加 1~2.5s(取决于模型与上下文长度)。

七、Token / LLM 网关

大模型与嵌入调用都经网关。底层复用公司已有的新网 xapi,底座只加一层薄中间件做应用维度的计量与治理(不重复造网关)。下面讲这两层的分工,以及随之而来的 Token 成本管控。

7.1 两层网关:复用底层 + 应用治理

公司已有新网 xapi 模型网关:一个 Key 接 OpenAI / Gemini / Claude 兼容的多模型(含 embeddings、rerank),统一鉴权与平台计费。但它是基础设施级、账户维度的——只到账户出账,分不清某次调用是哪个租户 / 业务 / 会话花的。底座因此不重复造模型接入,而是复用 xapi 做底层,在其之上加一层"应用维度网关",补齐它不做的事。

层由谁提供职责
底层模型接入复用新网 xapi(已有)统一接多模型(文本 / 嵌入 / 重排 / 图像…)、密钥托管、兼容协议、平台(账户级)计费
应用维度治理(薄中间件)底座新增按租户 / 业务 / 会话计量 token 与成本、配额限流、语义 + 结果缓存、按任务降级、调用审计 / PII 脱敏
为什么不再上一个完整网关(避免与 xapi 重复)
LiteLLM / Portkey 这类网关本身也做"统一多模型接入 + 路由 + 密钥",而这部分 xapi 已经做了——再叠一个就是"网关套网关",接入层重复。底座缺的只是 xapi 不做的应用维度计量与治理,所以这层做成一层薄中间件即可:优先自建(在编排服务里包一层,给每次 xapi 调用打 tenant/app/session 标签、从回包 usage 记 token,做配额与缓存);或退一步用 LiteLLM,但只把上游设成 xapi 这一个,纯借它的虚拟 Key / 预算 / 缓存,而不让它再做多厂商接入。

配置示例(应用层薄中间件,上游就是 xapi):

gateway:
  upstream: xinnet-xapi          # 底层模型接入,复用现有网关
  routes:
    - { model: cohere-embed-v4, use: embedding }
    - { model: claude-*, use: generation, fallback: gpt-* }
  metering: { by: [tenant, app, scene, session, task], record: tokens+cost }   # ← 多维计量,xapi 不做
  rate_limits: { tenant_default: { rpm: 600, tpm: 200000 } }
  budget:      { tenant: { monthly_usd: 5000, alert_at: 0.8 } }
  cache:       { semantic: true, ttl: 86400 }
  audit:       { log: all, pii_redact: true }

注:所选模型(如 Cohere Embed v4)若 xapi 暂未覆盖,应用层网关可直连对应厂商,或替换为 xapi 已提供的等价模型——路由对业务透明,不影响上层。

7.2 Token 经济(成本杠杆)

大模型与嵌入是主要可变成本。网关层把成本做到"按租户可见、按杠杆可控",主要省钱杠杆:

杠杆怎么省
增量跳过(content_hash)正文没变不重嵌入——离线最大头成本,增量场景绝大多数被跳过
语义 + 结果缓存重复/近似问答直接命中,不再调用嵌入与生成
按任务路由模型简单抽取/分类用小而便宜的模型,复杂生成才上大模型
重排压上下文rerank 后只喂 top8 而非 top50,生成 token 大幅下降
批量嵌入离线嵌入批处理,降单位成本
预算与配额按租户设月度预算与限流,到阈值告警/降级,杜绝失控

应用维度网关逐次记录 token 与费用,按租户 / 业务线出账——这正是新网 xapi(账户级)不做、需在应用层补上的;它既是成本控制,也是 RaaS 计价时的成本依据。

7.3 网关数据:计量、配额、缓存

计量按五个维度、由粗到细:租户 → 应用 → 场景 → 会话 → 任务。每条用量记录都带全链标签,于是可在任一层级聚合(roll-up),也可在多级同时设限(如:租户月度预算 + 应用日配额 + 会话/任务硬上限)。

这层是有状态的——用量要落库、配额要实时更新。表不多,但要建:

表 / 存储存什么更新时机选型
用量明细 usage每次调用带全维度标签:tenant / app / scene / session / task(trace)+ 模型 / op / prompt+completion tokens / 成本 / 延迟 / 是否命中缓存每次调用后追加写(append-only)Postgres 分区表(量大可转数仓聚合)
配额预算 budget多级:level(tenant/app/scene/session/task)+ 对象 + 预算/RPM/TPM 上限 + 实时已用每次调用后按各级累加 used;周期重置Postgres(强一致,实时累加)
缓存 cache语义缓存(查询向量→结果)、结果缓存(精确键→结果)命中即读、未命中写,带 TTLRedis(向量近似 + KV)
虚拟 Key(可选)租户 / 业务的 Key、权限范围、状态签发 / 吊销时更新Postgres

  "tenant_id":"t_zqdl", "app":"jianzhan", "scene":"detail_gen",

九、工程保障:非功能与治理

能跑起来只是第一步,还得扛得住、管得住。下面讲容量与性能、可用性、多租户、增量一致性、冷启动、失败降级,以及治理与安全。

9.1 容量与性能估算

先按量级设计、预留扩展;以下为占位估算,接入真实数据后校准。

维度估算(占位)
服务客户数数百 → 数千(按业务线增长)
单客户知识量数千 ~ 数万 chunk
全量向量规模千万级起步,架构预留十亿级
在线检索 QPS峰值数十 ~ 数百
延迟预算检索 P95 < 150ms;RAG 端到端 P95 < 3s(含生成)
存储估算向量 ≈ 维度 × 条数 × 4B + 原文/原始件入对象存储

9.2 非功能要求

  • 可用性:在线服务目标 SLA 99.9%;离线流水线允许有延迟,但必须可重跑、不丢数据。
  • 扩展性:在线服务无状态、水平扩展;向量库按租户/库分片;离线 Worker 按队列堆积自动扩缩。
  • 容灾备份:向量/图/文档库定期快照,对象存储跨区冗余;主数据(MDM)与 ACL 权限表优先级最高、独立备份;明确 RTO/RPO 目标。

9.3 多租户:设计与使用

  • 隔离方式:默认逻辑隔离——每条记录带 tenant_id,检索时强制注入租户过滤;大客户/敏感客户可升级为物理隔离(独立索引/命名空间/库)。
  • 防串户:权限在服务层兜底——任何检索都带租户与 ACL 上下文,A 客户数据不可能进入 B 客户结果。
  • 配额:网关按租户限流(RPM/TPM)+ 存储配额,单租户打满不影响他人。
  • 唯一例外:跨客户复用只允许脱敏后的领域库、效果库(按行业×市场切片)。
  • 租户模型:集团内业务线与外部客户都是租户;隔离统一落到 tenant_id 主线——向量库用 namespace、图库用标签/子图、文档库用行级 tenant_id。
  • 使用方式:调用方在网关注入租户上下文(token 带 tenant + role)→ 服务层据此下推过滤 → 计量计费按租户归集;接入时每条数据也打 tenant,做到"写入即归属"。

9.4 增量更新与一致性

  • 变更捕获:CRM/产品库等结构源用 CDC/Webhook 近实时触发,外部内容定时增量,只处理变化部分。
  • 原子切换:受影响 chunk 重新嵌入后写新版本,就绪再切换指针,旧版本保留以便回滚——检索侧不会读到半成品。
  • 写一致性:"文档库 + 向量库 + 图库"三写带补偿重试,最终一致 + 版本号兜底,失败可重放。
  • 失效下线:valid_to 到期或被标记失效的知识,检索时过滤、后台定期清理,避免旧知识误导。

9.5 冷启动

  • 新客户(无专属知识/无效果):先用行业库 + 领域库兜底(通用底料),客户库随交互逐步积累。
  • 新行业:先靠领域库的通用方法论,叠加快速采集该行业公开资料建"行业库种子"。
  • 效果库冷启动:没有本客户实测前,先借同行业×市场的切片经验做参考,跑出数据后切回本客户真实效果。

9.6 失败与降级

组件故障影响降级路径
向量库不可用语义检索失效降级到关键词检索(BM25)+ 命中缓存
图库不可用多跳推理失效GraphRAG 降级为纯向量 RAG
LLM 不可用无法生成答案网关切备用模型;仍不行则只返回检索片段、不生成
嵌入服务不可用入库/查询受阻离线入库排队等待;在线优先走缓存
重排不可用精度略降跳过 rerank,用原始检索分返回

9.7 治理与安全

  • 权限与安全:按租户/客户/角色分级,ACL 写在每条记录;跨客户复用只允许脱敏后的领域、效果知识。
  • 合规与数据驻留:出海数据按区域驻留、按授权使用,过程可审计。
  • 生命周期:知识带来源、时效、版本;过期下线或降权。

检索效果与真实成效(成交回流)沉淀进效果库、反哺优选,底座越用越准。具体质量度量按需做,本方案不展开。
